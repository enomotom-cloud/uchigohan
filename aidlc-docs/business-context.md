## 2. 競合分析

### 競合との差別化ポジション（連携前提でない自前領域での比較）

うちごはん **単体機能** で各専門サービスに勝てる領域は限定的です。

- 栄養データベースの精度では **あすけん** 等の専門サービスに及びません
- レシピの豊富さでは **楽天レシピ・クラシル** に及びません
- 食材調達の効率では **Amazon・Oisix** に及びません

これらの専門サービスは、それぞれの領域で長年の蓄積を持っています。
うちごはんが勝負するのは、これらの専門サービスがまだ手付かずの領域 ──
**「家族単位の "今日"」を会話と健康データから可視化する** こと ── です。

| サービス | 主な強み | うちごはんとの関係 |
|---|---|---|
| ナッシュ等冷凍宅配 | 栄養トータルマネジメント、完全代行 | 補完関係（料理を奪わない選択） |
| クラシル等レシピDB | 豊富なレシピ、UGC の多様性 | 連携を構想（API 公開時） |
| Oisix等定期便 | 計画的配送、食材の品質 | 補助救済としてリンク提供 |
| あすけん等栄養管理 | 栄養計算の専門性、膨大な食品 DB | 連携を構想（家族文脈での再利用） |
| 楽天レシピ | 公開 API、商用利用可 | API 経由で実装中（規約遵守） |
| **うちごはん（提案）** | **家族×今日の可視化と統合レイヤー** | （Inception 段階、MVP で検証予定） |

### うちごはんのユニークポジション

```
              「考える疲労」を消す
                     │
                  ◎──┼──────── うちごはん
                     │
        Oisix ────●──┤
                     │
            DELISH ●─┤
            クックパッド│
                     │
        ナッシュ ────────●         「作る楽しさ」を残す
        Uber Eats ──────●─→
                     │
                  ─────┘
            (作る楽しさを失う)
```

**うちごはんは右上の独自ポジション**：「考える疲労を消しながら、作る楽しさを残す」。既存サービスはこの両立をしていない。

## 3. レシピデータ戦略

### 採用：楽天レシピAPI（公式・無料・送客モデル）

**選定理由**：

| 観点 | AI生成レシピ（不採用） | 楽天レシピAPI＋AIキュレーション（採用） |
|---|---|---|
| 信頼性 | ハルシネーションリスク | 数百万件の実績レシピ |
| 法的責任 | 食中毒・アレルギーで責任曖昧 | 楽天が一次責任を負う |
| B2B信頼性 | 健保組合「AIが作った…？」 | 「楽天認定レシピをAIが選定」 |
| AI比率 | 過剰 | 適切（選定・栄養補正・家族最適化） |

### 楽天ウェブサービスAPI規約への準拠（重要）

**確認事項**（公式利用規約より）：

- **第10条(4)**：楽天アフィリエイト以外での収益化は原則禁止 → **本番商用展開時は楽天アフィリエイトプログラム加入必須**
- **第10条(9)**：取得情報を「不特定多数と共有可能な場所への保存禁止」 → DynamoDBでの私的キャッシュは可、公開は不可
- **第8条(4)**：楽天市場へのリンク必須、競合リンク禁止
- **第7条(3)**：リクエスト数制限の権利を楽天が留保 → 短時間大量同一リクエストは制限される可能性
- **第12条**：ユーザー削除要求への即対応必須

楽天レシピ API 規約第 10 条(4)では、楽天アフィリエイト以外での
収益化が禁止されています。これは規約上の制約として明確であり、
Phase 3 段階で楽天アフィリエイト PG への加入手続きを行うことで
規約遵守を確保します。

### キャッシュ戦略

- **TTL：3時間**（保守的設定。1秒1リクエスト体感制限の回避と規約準拠の両立）
- 24時間以上のキャッシュは規約解釈上グレーのため避ける
- ユーザーごとの先読み（朝の事前生成）で体感速度を確保
- 削除要求時は関連キャッシュも即削除

### Phase別の楽天API活用

- **Phase 1 (MVP/予選)**：個人開発・非商用デモとして利用、書類審査・予選デモは規約問題なし
- **Phase 2 (決勝)**：同上、デモ目的範囲
- **Phase 3 (事業化)**：**楽天アフィリエイトプログラム加入必須**、楽天市場への送客手数料がレベシェの一部に

### 補助：Spoonacular（Phase 2 多言語対応時）

海外向け展開時に検討。日本語精度△のためPhase 1では不採用。

## 4. ビジネスモデル：B2B2C福利厚生

### 想定する売り先と単価

| 売り先 | なぜ買うか | 想定単価 | 営業難易度 |
|---|---|---|---|
| **健康保険組合** | 健康経営、医療費削減、介護予防 | 月500-1000円/従業員 | 中 |
| **大企業福利厚生部門** | 共働き支援、ビジネスケアラー支援、離職防止 | 月300-800円/従業員 | 中 |
| **自治体** | 高齢者見守り、介護予防、フードデザート対策 | 補助金スキーム | 高 |
| **食品宅配会社（Oisix・パルシステム等）** | 既存サービスへの上乗せ機能 | レベシェア15-20% | 低 |
| **楽天アフィリエイト** | 楽天市場・楽天レシピへの送客 | 取引手数料2-4% | 低（PG加入のみ） |
| **栄養士の副業マーケット** | アドバイス課金の手数料収入 | 取引手数料15-20% | 低 |

### 収益構造

```
主収益（70%）：B2B SaaS月額課金
  - 健康保険組合・大企業向け
  - 1社あたり500-2,000名規模、ARR 600万-2,400万円

副収益（25%）：食品EC連携 + 楽天アフィリエイト
  - Amazon Fresh、Oisix等への送客手数料
  - 楽天市場・楽天レシピへの送客（規約準拠）
  - ユーザーあたり月300-500円相当

副収益（5%）：栄養士マーケット手数料
  - 個別アドバイス購入の取引手数料
  - 単価3,000-10,000円/相談
```

### B2B営業向け価値計算（為替訂正済み）

**コスト試算**（2026年5月時点）：

| 項目 | 月額 |
|---|---|
| Bedrock推論（Sonnet+Haiku） | $1.5/月 = **¥240/月**（1USD=¥160、2026年5月時点） |
| Lambda + DynamoDB + その他AWS | ¥30/月 |
| LINE Messaging API（本番ライトプラン） | ¥5,000/月 ÷ 利用者数 ≒ ¥5/月（1,000人時） |
| **合計** | **¥275/月程度** |

**粗利試算**：

| プラン | 月額収入 | 月額原価 | 月額粗利 | 粗利率 |
|---|---|---|---|---|
| 月額500円プラン | ¥500 | ¥240 | ¥260 | **52%** |
| 月額800円プラン | ¥800 | ¥240 | ¥560 | **70%** |

> 注：原価は推論コスト中心。LINE有料プラン費は規模に応じて分散するため上記には含まず。  
> 為替レートは1USD=¥160、2026年5月時点の参考値。

**従業員1名の年間効果**：

```
献立を考える時間: 平均20分/日 × 365日 = 122時間/年
時給換算（平均年収500万・年間労働時間2,000h）: 2,500円/h
→ 約30万円/年の認知労働コスト削減

うちごはんの月額: 500-800円 × 12ヶ月 = 6,000-9,600円/年
→ 投資対効果 30倍以上
```

### 連携合意に関する Inception 段階の位置づけ

本書類段階でうちごはんが「連携」を語る各サービス（あすけん、Oisix、
健保組合等）について、**合意は取得していません**。

これは Inception フェーズの性質上、構想と検証可能性の提示が主目的で
あり、合意取得は事業フェーズ移行後の個別交渉に委ねるためです。

うちごはんが先に証明すべきは：

1. **家族単位の "今日"** を会話と健康データから可視化できること
2. その可視化が **ユーザーにとって価値ある提案** につながること
3. その提案が **既存の専門サービスにとっても顧客接点として価値がある** こと

この 3 点が証明されて初めて、各専門サービスとの連携交渉が成立する
構造です。書類段階での「連携します」と書く傲慢を避け、
「連携可能な構造を目指します」と書きます。

## 5. LINE Messaging API 料金プラン（公式・確認済み）

| プラン | 月額 | 無料メッセージ通数 |
|---|---|---|
| コミュニケーションプラン（Free） | 0円 | 200通/月 |
| ライトプラン | 5,000円 | 5,000通/月 |
| スタンダードプラン | 15,000円 | 30,000通/月（追加〜3円/通） |

**重要**：
- リプライメッセージ（ユーザー発言への返信）は**通数カウント外**で実質無料
- メッセージ通数は「送信対象人数」でカウント（メッセージオブジェクト数ではない）

### Phase別のプラン

- **Phase 1〜2（ハッカソンデモ）**：**コミュニケーションプラン（無料・200通/月）**で十分。デモ参加者数名×数十回のテストなら問題なし
- **Phase 3〜4（本番運用）**：**ライト or スタンダードプランへ移行**。Bユーザー社員数1,000名規模であればライトプラン（5,000通/月、5,000円）が現実解
- **規模拡大期**：プッシュメッセージ最適化（リプライ駆動を最大化、プッシュは退勤前など必要最小限）でコストコントロール

## 6. Phase別ロードマップ（事業視点）

### Phase 1: ハッカソン書類審査〜予選（〜2026/5/30）
- LINE Bot コミュニケーションプラン（無料）
- MVP完成（U1-U4, U7, U9, U10）
- 楽天レシピAPI（個人開発・非商用デモ）
- 想定ユーザー：自チーム＋10名のクローズドβ
- KPI：MVPでメニュー受信フローが20秒以内に完結すること

### Phase 2: ハッカソン決勝（〜2026/6/26）
- 親等モデル全機能実装（U5-U6, U8追加）
- 想定ユーザー：50名のクローズドβ（家族単位）
- KPI：家族メンバー間の閲覧が誤公開ゼロで成立

### Phase 3: 事業化準備（2026下期）
- LINE Bot ライトプランへ移行
- **楽天アフィリエイトプログラム加入**（必須）
- 1社目のパートナー獲得（Oisix等の食品ECか健保組合）
- 法務確認（要配慮個人情報、医師法、家族同意UI）
- KPI：有償ユーザー1,000名

### Phase 4: 本格展開（2027〜）
- 健康保険組合向け正式リリース
- LINE Bot スタンダードプラン
- 楽天市場本格連携
- 栄養士マーケットローンチ
- KPI：MAU 10万人、ARR 5億円

## 7. リスクと対策

### 法的リスク

| リスク | 影響 | 対策 |
|---|---|---|
| 健康診断結果の取扱い | 個人情報保護法（要配慮個人情報） | オプトイン明示、Comprehend Medicalで匿名化、削除権担保 |
| 医師法（無資格診療） | 月次レポートが医療助言とみなされるリスク | 「傾向把握」表現に統制、診断・処方助言は禁止、「気になったら主治医へ」明記、ガードレール層 3（[医療助言禁止フレーズ統制](./03-application-design.md#ガードレール設計)）で出力後再チェック |
| 食品衛生（レシピの安全性） | アレルギー誤判定での健康被害 | 楽天レシピへ送客（楽天が一次責任）、4 層ガードレール設計（[03-application-design.md](./03-application-design.md#ガードレール設計) 参照）：プロフィール参照／ホワイトリスト／医療助言フレーズ統制／Bedrock Guardrails（Phase 2 構想） |
| 家族同意なしの監視 | プライバシー侵害 | 親等別公開は双方向承認必須、未成年は親権者承認 |
| 楽天ウェブサービス規約 | 商用展開時の規約違反リスク | Phase 3でアフィリエイトPG加入、楽天市場送客必須化 |
| LINE Messaging API利用規約 | スパム判定・凍結リスク | プッシュメッセージはオプトイン明示、ユーザー単位の頻度制御 |

### 倫理リスク

| リスク | 対策 |
|---|---|
| 「監視されている」と親世代が感じる | 親側で公開範囲を完全コントロール、デフォルト最小公開 |
| AIへの過度な依存で料理スキル低下 | U10をAI 0%に固定、楽天レシピで「教える」を担保 |
| 家族不在時の「孤食」サポート | 一人の食事も価値あるものとして扱うUI設計 |

### 技術リスク

| リスク | 対策 |
|---|---|
| 食材認識精度が80%に届かない | Bedrock Vision を主、Rekognitionは補助、手動修正UI |
| Bedrockコスト増大 | プロンプトキャッシュ、候補メニューの非同期事前生成、ユーザー単位レート制御 |
| 親世代のLINE操作困難 | 写真撮るだけUI、設定は子が代行可、Phase 2でPolly音声操作 |
| 家族関係の複雑性（離婚・再婚等） | 親等の柔軟な設定、関係性の手動上書き可 |
| LINE申請遅延 | 独自Webチャット（CloudFront+S3+Lambda）をフォールバック準備 |
| 楽天レシピAPI制限超過 | TTL 3hキャッシュ、朝の先行生成、レート制御 |

### 事業リスク

| リスク | 対策 |
|---|---|
| 健保組合・大企業の意思決定が遅い | 食品EC連携を先行収益源として確保 |
| 既存プレイヤー（Oisix等）のAI機能追加 | Oisix側のアップサイド機能として組み込まれるBizDev戦略も視野 |
| ナッシュ等の値下げ競争 | 「楽の外注」ではなく「愛情を守る」ポジションで差別化を維持 |
| クックパッド等のAPI公開 | 公開時はマルチソース対応で機能拡張、競合ではなく協業 |

## 9. 「人をダメにする」テーマへの最終回答

> AI完全代替時代の今、私たちは「**何でもAIで代替できる**」ことを発見しました。  
> だからこそ、うちごはんは問いかけます。  
> 「**代替してはいけないものは何か？**」  
>   
> 答え：**手料理に込める愛情です。**  
>   
> 私たちは「考える疲れ」だけをAIに渡し、「作る愛情」を人間に残します。  
> サボれる人ほど、家族を大事にできる。  
> これが、うちごはんが提案する「**良い怠惰**」です。  
>   
> ── うちごはんは、料理を奪いません。  
> ── うちごはんは、思考だけを奪います。  
> ── それでも限界の日には、最後の救済（Tier 4）を用意しています。
